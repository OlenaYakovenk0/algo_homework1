## 1. Структури даних (для операцій V2)

Три реалізації під єдиний інтерфейс:
- **V1:** `vector<Student>`; пошук і зміни — лінійні.
- **V2:** `unordered_map<email, Student>`; зміна групи за email — O(1) у середньому; пошук за ДН — лінійний обхід.
- **V3:** `map<(month,day), vector<Student>>` для швидкого пошуку за ДН; зміна групи — лінійно по відповідному списку.

## 2. Тестування

- **Дані:** N ∈ {100, 1e3, 1e4, 1e5}.  
- **Навантаження:** випадковий мікс трьох операцій протягом **10 с**.  
- **Метрики:** загальна кількість операцій за 10 с; використання пам’яті (MB).  

### 2.1. Пам’ять

Тренди: **V1** — найекономніший; **V2** — помірне зростання; **V3** — найбільша пам’ять через індекс за датою народження.

### 2.2. Продуктивність (операцій за 10 с)

Тренди: зі зростанням N throughput падає у всіх; **V1** трохи кращий на малих N; **V2** тримає стабільний компроміс; **V3** повільніший через додаткові структури та оновлення.

**Висновок по V2:** найкращий баланс для цього набору операцій: швидкі оновлення за email і прийнятна пам’ять; лінійний пошук за ДН тут достатній.

## 3. Завдання III — сортування (S5)

- **Метод 1:** `std::sort` за `m_group`.  
- **Метод 2:** власний **Radix Sort** (стабільний; фіксована довжина/алфавіт).
